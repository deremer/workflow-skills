The Anatomy of a Workflow Skill

How to turn a repeatable, multi-step process into an agentic workflow that stops for human review, hands work between phases on explicit contracts, and checks itself before it asks you to look.

The problem

A single prompt is the wrong shape for multi-step work

Hand an agent a five-step job in one prompt and it will usually do something reasonable and quietly wrong. The failure modes are predictable, and they repeat across tasks and models.

  • It skips steps it judges unimportant.
  • It fabricates the handoff between steps, inventing inputs the previous step never produced.
  • It loses early decisions as the context fills, and the problem gets worse the longer the task runs.
  • It satisfies a research step with two quick searches instead of the depth the work needs.
  • It revises forever, or it stops too early, with no agreed limit either way.

These are structure failures more than model failures. The fix is to give the work a shape. Distinct phases, explicit handoffs, and a human checkpoint between them. That shape is a workflow skill, and the rest of this guide is about what makes one hold up.

The test

Not every task deserves a workflow

A workflow skill earns its complexity only when four things are true at once:

  • The job has three or more distinct phases that produce different artifacts.
  • A human needs to review the result between phases.
  • Each phase depends on the one before it.
  • The process repeats often enough to be worth encoding.

If a single skill or a single prompt would do, use that instead. The skill itself runs this check in its first phase and will tell you to simplify when a workflow is overkill. When all four hold, you have a real workflow, and the design choices below start to matter.

Architecture

A thin conductor, with the detail kept out of sight

A workflow skill splits into two layers. A thin orchestrator, the SKILL.md file, names the phases, the gates, and the rules. The detail for each phase lives in its own reference file, read only when that phase runs. This is progressive disclosure. The agent loads the map up front and the territory on demand.

Keeping the orchestrator thin is what lets the agent hold the whole workflow in view at once, without any single phase's detail crowding out the others. The orchestrator never carries phase-level questions, research instructions, or output formats. Those belong in the phase file, where they load at the moment they are needed.

Quality control · Gates

The work stops and waits for you

A gate is a hard stop at the end of a phase. The agent presents what it produced, states what you should evaluate, and waits. It does not drift into the next phase while you are not looking. Every gate in a generated workflow carries the same four prohibitions:

  1. Stop. Wait for explicit approval before any next action.
  2. Do not read the next phase's reference file.
  3. Do not begin work on the next phase.
  4. Do not summarize what the next phase will do.

The language is deliberately absolute. Never skip a gate. A gate written as a suggestion is not a gate. Each gate also carries a focus, the specific thing you are meant to judge at that point. Source quality after research. Tone and structure after a draft. Brand compliance after formatting. A good gate tells you what to look at, not just that something is ready.

Quality control · Contracts

Every handoff is a named contract

The weakest point in any multi-step process is the handoff. A workflow skill makes each handoff explicit with input and output contracts. A contract names the artifact, its format, and its internal structure. Not "research findings from phase 1" but "approved research brief, markdown, with Summary, Sources with URLs, Key Findings tagged FACT, PROJECTION, or OPINION, and Gaps."

Then the chain gets verified. For every transition, the previous phase's output must produce every field the next phase's input expects, in a compatible format. If phase two emits markdown and phase three wants JSON, that is a gap, and it gets fixed before the workflow ships. Specific contracts are what let a later phase trust an earlier one without re-reading its work.

Quality control · Validation

A phase checks itself before it asks you

Before a phase presents at its gate, it validates its own output against a checklist, fixes what fails, and re-runs the whole checklist from the top. It keeps looping until every check passes. Only then does it reach the gate. This is the difference between a checklist you read once and a loop you run until it is green.

Re-running from the top matters because a fix can introduce a new problem a one-pass check would miss. The same idea drives the build phase of the skill itself, which audits generated files for anti-patterns, fixes them, then re-verifies the contract chain from the start, because a fix to one phase's output can break the next phase's input.

Quality control · Depth

Research that cannot be satisfied with two searches

Left to its own judgment, an agent will declare a research task done after a couple of searches. An execution depth contract removes the judgment. It sets numeric floors. A minimum number of searches. A minimum number of full sources read rather than skimmed. A minimum number of unique sources cited. It tells the agent to start broad and narrow, to chase the most promising sources to full text, and to label every claim as fact, projection, or opinion.

The rule that matters most

State gaps explicitly. Never fill a gap with plausible-sounding content. A workflow that invents a number to look complete is worse than one that says it does not know.

Depth is calibrated to how often the workflow runs. A daily report does not need fifteen searches per run. A one-time strategy document might. The skill matches depth and revision limits to frequency, and it will override a request for heavy research on a daily workflow when the token economics do not justify it.

Execution

Where each phase runs, and why context stays clean

Not every phase should run in the same place. A workflow skill chooses an execution model per phase. Light work runs in the main session. A research phase that needs many searches runs in an isolated subagent, so its raw results never crowd the orchestrator's context, returning only a structured brief. Work that needs several independent research streams cross-referenced runs as a team of agents.

Two more levers. Model choice per phase, with a cheap, fast model for broad exploration and a stronger model for synthesis and architecture. And portability, where subagents and forked context are enhancements on platforms that support them, with a sequential fallback so the same workflow still runs where they do not. Keeping the orchestrator's context lean is what lets a long workflow survive without losing its earliest decisions.

What goes wrong

The failure modes a good workflow designs out

Most broken workflows fail in the same handful of ways. The skill audits for each one and fixes it before delivery. Knowing the smells makes you a better reviewer of your own designs.

Mega-skill smell

One giant SKILL.md holding every question, format, and instruction.

Fix: move detail into phase files and keep the orchestrator thin.

Implicit handoff smell

A phase ends and the next begins with no gate in between.

Fix: every phase ends with a hard stop and the four prohibitions.

Optional gate smell

"Review this if you'd like." A gate phrased as a suggestion.

Fix: gates use mandatory language only.

Context amnesia smell

By the last phase, the early decisions are gone.

Fix: a workflow-state block that carries decisions and the reasoning behind them.

Unbounded loop smell

"Refine until it's good." No limit on revisions.

Fix: a specific revision count per phase.

Assumed input smell

A phase describes what its input "should" contain.

Fix: name the actual prior artifact, its format, and its structure.

Trigger collision smell

A trigger phrase that also matches a common, unrelated skill.

Fix: make the phrases specific to this workflow.

Shallow research smell

A depth contract that says eight searches over a process that describes three.

Fix: enough discrete search steps to meet the floor.

Prose question smell

Open prompts where the answers are predictable.

Fix: structured options with an escape hatch.

Platform-blind smell

Forked context or hooks with no fallback for platforms that lack them.

Fix: a sequential path that runs everywhere.

State

A workflow that can be picked back up

A long workflow will outlive the context window that started it. So each gate produces a compact state block: which phase completed, the key decisions, the artifacts produced, the open questions, and the next phase. The field that matters most is decision rationale, the why behind each choice.

A summary can compress what was decided. Only the rationale lets a later phase, or a resumed session, understand whether a decision still holds. Resumption is built in. Read the most recent state block, confirm it with the user, continue from the next phase, and do not re-read the phases already done. The state block is the workflow's memory, written to survive being summarized.

The principle

What this teaches about building agents

Strip away the specifics and a workflow skill is a set of bets about how to get reliable work from a capable but inconsistent agent. The bets generalize past this one skill.

01

Front-load planning

The cheapest place to fix a mistake is before any work is done. Spend the effort on scope and design.

02

Gate autonomy with review

Let the agent run, but put a human checkpoint wherever being wrong is expensive to undo.

03

Make handoffs contracts

Name what moves between steps, its format, and its structure. Vagueness is where multi-step work breaks.

04

Bound every loop

Revision, research, retries. A loop without a limit is a way to burn time or to stop too soon.

05

Verify before declaring done

A self-check that runs until it passes beats a confident claim of success.

06

Capture decisions so the system compounds

Write down the why. The next run, and the next person, inherit it instead of relearning it.
